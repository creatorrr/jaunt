# claude-codex-shim: run Jaunt's Codex engine through the Claude CLI

Jaunt's only generation engine drives the `codex exec` subprocess. In
environments without `codex` (no install, no `codex login`) but with an
authenticated `claude` CLI, this skill's shim stands in: a script named
`codex` that accepts Jaunt's invocation and answers over the same JSONL
protocol, with `claude -p` doing the actual model call.

## Install (per session)

```bash
chmod +x .claude/skills/claude-codex-shim/bin/codex   # once per checkout
export PATH="$PWD/.claude/skills/claude-codex-shim/bin:$PATH"
```

Verify before relying on it:

```bash
echo 'Reply with exactly: PONG' | codex exec -C /tmp --sandbox workspace-write -m gpt-5.5 --json -
# → {"type": "item.completed", "item": {"type": "agent_message", "text": "PONG"}}
# → {"type": "turn.completed", "usage": {...}}
```

Then run Jaunt commands normally (`jaunt reconcile`, `jaunt build`, …) with
that PATH active.

## What Jaunt actually consumes (why this works)

`run_codex_exec` in `src/jaunt/generate/codex_backend.py` sends the prompt on
stdin and parses only these JSONL events from stdout:

- `item.completed` with `item.type == "agent_message"` → the final text
- `turn.completed` with `usage.{input_tokens, output_tokens, cached_input_tokens}`
- `turn.failed` / `error` → failure message

The shim maps `claude -p --output-format json` onto exactly that
(`cache_read_input_tokens` → `cached_input_tokens`). Everything else on the
codex command line is accepted and ignored, except `-C <dir>`, which becomes
the Claude working directory plus `--add-dir` (edit permission scoped to the
workspace, approximating `--sandbox workspace-write`).

## Limits and caveats

- **Model flags are ignored.** `-m gpt-5.5` and `model_reasoning_effort` don't
  map; the shim uses the Claude CLI's default model. The `[semantic_gate]`
  small-model economy (gpt-5.4-mini) silently becomes a full-model call.
- **Provenance.** This repo's policy pins real builds to Codex/gpt-5.5. The
  shim is for dev-loop work: exercising model-calling paths end-to-end,
  deriving throwaway batteries, demos. Do NOT use it to regenerate committed
  self-hosted artifacts (`src/jaunt/__generated__/**`, `.pyi` stubs,
  `tests/contract/jaunt/**`) — that would commit Claude-generated bytes under
  a gpt-5.5-labelled pipeline.
- **`codex exec` only.** `codex mcp-server` / `codex app-server` are not
  emulated; unknown subcommands exit 2.
- **Permissions.** `claude -p` runs with `--permission-mode acceptEdits`
  scoped to the workspace dir. File-writing build tasks work, but there is no
  OS-level sandbox — don't point it at directories you don't want edited.
- **Latency/cost.** One `claude -p` process per generation call, no prompt
  cache reuse across calls beyond what the API gives you. Fine for a handful
  of calls; slow for a big `jaunt build` fan-out.

## Proven end-to-end (2026-07-07)

Live run on a demo contract project: `jaunt reconcile` derived a battery whose
prose `Properties:` bullet was transcribed by Claude into a rooted invariant,
`jaunt check` then passed offline (no shim needed), and a body edit flipped
`check` to exit 4 `behavior-drift`. See
`docs/superpowers/specs/2026-07-07-contract-properties-design.md`.
