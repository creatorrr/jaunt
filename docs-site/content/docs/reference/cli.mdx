For a guided, end-to-end introduction, start with [Quickstart](/docs/quickstart).

Entry point: `jaunt = "jaunt.cli:main"` (see `pyproject.toml` in the repo root).

All generation commands run through the Codex engine
(`[agent].engine = "codex"`, the only supported value). They require the
`codex` CLI installed and authenticated (`codex login`).

## Init

Create a starter `jaunt.toml` in the target directory:

```bash
uv run jaunt init
uv run jaunt init --root /tmp/myproj
uv run jaunt init --force
uv run jaunt init --json
```

## Build

Generate implementation modules for `@jaunt.magic` specs:

```bash
uv run jaunt build
uv run jaunt build --force
uv run jaunt build --jobs 16
uv run jaunt build --target my_app.specs
uv run jaunt build --no-infer-deps
uv run jaunt build --json
```

If an upstream module's exported API changed, `jaunt build` also regenerates its stale dependents.

Generation-shaping flags (also accepted by `jaunt test` and `jaunt watch`):

- `--instruction "..."`: extra build instruction appended to the prompt (repeatable).
- `--include-target-tests` / `--no-include-target-tests`: include targeted test-spec source in build prompts (default: off, or `[build] include_target_tests`).
- `--no-auto-skills`: skip auto-generated PyPI skills for this run.
- `--no-builtin-skills`: don't seed Jaunt's bundled builtin skills into the Codex workspace.
- `--no-repo-map` (build only): disable repo-map injection for this build. See [Repo Context](/docs/reference/repo-context).
- `--no-semantic-gate` (build/test): force every normalized-digest change to rebuild, skipping the Layer B semantic gate (Layer A linter-resistance still applies). See [Change Detection](/docs/reference/change-detection).

## Test

Generate tests for `@jaunt.test` specs and run pytest:

```bash
uv run jaunt test
uv run jaunt test --no-build
uv run jaunt test --no-run
uv run jaunt test --pytest-args=-k --pytest-args email
uv run jaunt test --no-redact-derived   # debug only — see below
uv run jaunt test --json
```

Generated tests are split into a **held-out set**: cases anchored to the docstring's
canonical examples are tagged `@pytest.mark.jaunt_tier("example")`, everything the test
generator derives itself is tagged `@pytest.mark.jaunt_tier("derived")`. When a test
fails and Jaunt regenerates the implementation, *derived* failures are redacted to an
opaque id + exception class (the implementation generator never sees the expected value
it would otherwise game), while *example* failures keep full detail. `--no-redact-derived`
disables this and feeds full failure detail back for all tiers — a **debugging escape
hatch only**; it defeats the implementer/tester barrier and logs a warning.

## Contract Mode

Adopt existing, committed code under docstring contracts and keep a derived
pytest battery in sync (see the [Contract Mode guide](/docs/guides/contract-mode)):

```bash
uv run jaunt adopt my_app.slug:slugify   # add @jaunt.contract + derive its battery
uv run jaunt reconcile                   # derive/refresh committed batteries (calls the model)
uv run jaunt check                       # verify batteries deterministically (CI gate, no model)
uv run jaunt eject my_app.slug:slugify   # remove contract tracking; leave plain pytest
uv run jaunt eject --all
```

`jaunt reconcile` is the only contract command that calls the model. `jaunt check`
is deterministic and exits `4` on any blocking drift state (unbuilt /
stale-prose / signature-drift / behavior-drift), which makes it the CI gate.

## Status

Show stale/fresh modules:

```bash
uv run jaunt status
uv run jaunt status --target my_app.specs
uv run jaunt status --magic-only
uv run jaunt status --json
```

`jaunt status` should reflect the same dependency-driven freshness model as `jaunt build`, including modules invalidated by an upstream dependency API change. `--magic-only` probes only `@jaunt.magic` freshness, skipping contract checks and repo-map drift.

## Specs

List `@jaunt.magic` specs and their dependency graph (useful for agents and tooling):

```bash
uv run jaunt specs
uv run jaunt specs --module my_app.specs
uv run jaunt specs --no-infer-deps
uv run jaunt specs --json
```

## Instructions

Print a tight, **project-aware** agent primer to load into a coding agent's
context before it operates Jaunt:

```bash
uv run jaunt instructions
uv run jaunt instructions --json
```

The output is the framework rules (the two authoring modes, the build/test loop,
how to write a good spec, the command and exit-code reference) followed by a live
snapshot of the current project: resolved `[paths]`, engine/model, semantic-gate
and repo-map settings, and which modules are stale. It ships with the package, so
the briefing always matches the installed version.

Run outside an initialized project, it still succeeds (exit `0`) — it prints the
framework rules plus a note to run `jaunt init`. `--json` emits
`{command, ok, text, project}`, where `project` is the structured live snapshot
(or `null` when there is no `jaunt.toml`).

<Callout>
  `jaunt instructions` is the single source of truth for the agent workflow. The
  bundled Claude Code / Codex skills are thin stubs that point here, so they
  cannot drift from the installed CLI.
</Callout>

## Tree

Maintain `treedocs.yaml`, the one-line-per-path repo map injected into build
prompts (see [Repo Context](/docs/reference/repo-context)):

```bash
uv run jaunt tree             # sync the tree (AST-only descriptions by default)
uv run jaunt tree --check     # CI gate: exit 4 if the tree is stale
```

LLM enrichment of descriptions is controlled by the `[context] enrich` config
flag (default `false` — AST-only, offline).

## Log

Tail the `JAUNT_LOG` change journal that build/test/daemon runs append to:

```bash
uv run jaunt log
uv run jaunt log -n 50          # 0 = all lines
uv run jaunt log --module my_app.specs
uv run jaunt log --json
```

## Clean

Remove generated directories under configured source/test roots:

```bash
uv run jaunt clean
uv run jaunt clean --dry-run
uv run jaunt clean --json
```

## Watch

Watch source/test roots and rebuild on relevant `.py` changes:

```bash
uv run jaunt watch
uv run jaunt watch --test
uv run jaunt watch --json
```

When `--json` is enabled, watch emits one JSON object per rebuild cycle (the watch envelope), not nested build/test JSON documents.

## Daemon Jobs

Run background codegen jobs from committed spec changes:

```bash
uv run jaunt daemon start
uv run jaunt daemon status
uv run jaunt daemon stop

uv run jaunt jobs
uv run jaunt jobs show <id> --full
uv run jaunt jobs retry <id>
uv run jaunt jobs land <id>
uv run jaunt jobs land --all
uv run jaunt jobs discard <id>
uv run jaunt jobs wait --timeout 1800
uv run jaunt jobs wait <id> --timeout 1800 --json --progress plain
```

`jaunt jobs` shows job records, pending rebuilds, and the latest heartbeat phase for
running jobs. `jaunt jobs wait` blocks until a target job finishes, or without an id
until the daemon is idle; `--settle N` controls the idle race-guard window and
`--timeout N` exits with code `5` if time runs out. A parked proposal (`PROPOSED`)
is treated as terminal-green, so `jaunt jobs wait` returns `0` on it.

Under the default propose-only mode (`[daemon] auto_commit = false`), green jobs
are parked as proposals rather than committed:

- `jaunt jobs land <id>` lands one parked proposal as a provenance commit. It
  re-validates first: a stale spec digest, a mismatched branch, dirty target
  paths, or a 3-way apply conflict all refuse the land (exit `4`); a stale digest
  or apply conflict also marks the proposal `SUPERSEDED`. There is no `--force`.
- `jaunt jobs land --all` lands every fresh proposal in job-creation order (the
  daemon queues in dependency order), prints per-job outcomes, and exits `0` only
  when every attempted land succeeded, else `4`.
- `jaunt jobs discard <id>` marks a proposal `DISCARDED` and removes its patch
  artifact.

Bad ids or wrong-state jobs exit `2`.

<Callout>
  **Automation**

  | Pattern | Command |
  |---------|---------|
  | Blocking agent loop (propose-only) | `git commit … && uv run jaunt jobs wait --timeout 1800 && uv run jaunt jobs land --all` |
  | Blocking agent loop (auto-commit) | `git commit … && uv run jaunt jobs wait --timeout 1800` |
  | Push notification | Set `[daemon] notify_command` to run your hook on proposed, landed, parked, or failed jobs. |
</Callout>

See the [Background Daemon guide](/docs/guides/daemon) for the full workflow.

## Guard

A PreToolUse hook for coding agents that warns when a tool call is about to
touch files under the generated dir:

```bash
uv run jaunt guard
uv run jaunt guard --generated-dir __generated__
```

It reads the hook payload on stdin and emits a warning for `__generated__`
paths (see `docs/hooks.md` in the repo for Claude Code wiring).

## Cache

Inspect or clear the LLM response cache under `.jaunt/`:

```bash
uv run jaunt cache info
uv run jaunt cache clear
```

## Skills

Manage project-local skills under `.agents/skills/`:

```bash
uv run jaunt skill list
uv run jaunt skill show rich
uv run jaunt skill add rich --description "Rich usage notes" --lib rich
uv run jaunt skill remove rich -f
uv run jaunt skill import --from /path/to/skills-dir
uv run jaunt skill refresh
uv run jaunt skill build rich
```

Two skill commands generate content through the Codex engine:

- `jaunt skill build <name>`: expand or refine a user-managed checked-in skill
- `jaunt skill refresh`: refresh Jaunt-managed auto-generated skills

Like `build` and `test`, both drive `codex exec` under the hood.

Note: `jaunt skill build <name>` expects the skill to exist already and to have
library metadata, usually from `jaunt skill add <name> --lib <package>`.

## Common Flags

- `--root /path/to/project`: override project discovery (otherwise Jaunt searches upward for `jaunt.toml`).
- `--config /path/to/jaunt.toml`: override config path.
- `--target MODULE[:QUALNAME]`: restrict work to one or more modules (currently module-level; `:QUALNAME` is ignored for filtering).
- `--no-infer-deps`: disable best-effort dependency inference (explicit `deps=` still applies).
- `--progress {auto,rich,plain,none}`: progress output mode; `auto` is rich on TTY and plain lines off-TTY. Explicit `--progress plain` works with `--json` (progress on stderr, JSON on stdout).
- `--json`: emit structured JSON output for agent/CI workflows.

## Exit Codes

- `0`: success
- `2`: config/discovery/dependency-cycle errors
- `3`: generation errors (LLM/backend/validation/import)
- `4`: pytest failure, contract check/reconcile block, or daemon job failed/parked during `jobs wait`
- `5`: timeout while waiting for daemon jobs

Note: `jaunt test` runs pytest only on the generated test files it just wrote (not the entire suite). Run `pytest` separately for a full test run.
