Jaunt maintains a lightweight map of your repository and injects it into build
prompts, so the generator knows what exists around the module it's writing.
It's on by default and works offline.

## The Repo Map (`treedocs.yaml`)

`treedocs.yaml` at the project root holds a one-line description of each
directory and `.py` file. Descriptions come from an AST baseline (module
docstrings, top-level symbols) — deterministic and offline — with opt-in LLM
enrichment.

```bash
jaunt tree             # sync the tree incrementally (AST-only by default)
jaunt tree --check     # CI gate: exit 4 if the tree is stale (new/ghost paths or edited files)
```

Whether descriptions get LLM enrichment is controlled by `[context] enrich`
(default `false` — AST-only, offline).

`jaunt build` keeps the tree in sync automatically and renders a repo-map block
(capped at `[context] max_chars`) into every build prompt. Disable it for one
build with `jaunt build --no-repo-map`, or entirely with `[context] repo_map = false`.
Source digests are cached in `.jaunt/tree-cache.json`, so syncs only touch what
changed.

## Configuration

```toml
[context]
repo_map = true             # maintain treedocs.yaml + inject a repo map into build prompts
repo_map_file = "treedocs.yaml"
enrich = false              # opt-in: LLM-enrich descriptions (else AST-only, offline)
max_chars = 6000            # cap the injected repo-map block
overview = false            # opt-in: model-written architecture overview (see below)

[context.search]
enabled = false             # opt-in; requires the `colgrep` binary on PATH
internal_retrieval = true   # Jaunt queries `colgrep --json` and seeds _context/relevant_*.py
max_hits = 8
```

## Semantic Retrieval (colgrep, Opt-In)

With `[context.search] enabled = true` and the `colgrep` binary (LightOn
next-plaid) on your `PATH`, Jaunt runs semantic retrieval before each build:
it queries `colgrep --json` for code relevant to the spec being built and seeds
the top `max_hits` results into the workspace as `_context/relevant_*.py`, so
the generator can read real neighboring implementations. If `colgrep` is
missing, Jaunt falls back gracefully — the build proceeds without retrieval.

## Project Overview (Opt-In)

`[context] overview = true` adds a model-written architecture overview of the
whole codebase to build prompts. Jaunt calls the model once — when the spec
sources, repo map, injected project docs (README/AGENTS/CLAUDE), or overview
prompt templates change — and caches the prose to `.jaunt/PROJECT_OVERVIEW.md`
(digest-keyed), so subsequent builds reuse it. The call is charged against
`[llm] max_cost_per_build` and shown in the cost summary. Toggling the flag
participates in build freshness, so enabling it triggers a one-time rebuild of
already-built modules.

## The Jaunt Preamble

Every build prompt opens with a static preamble
(`src/jaunt/prompts/codex_preamble.md`) that frames what Jaunt is and states
the signature/docstring contract — including the core design laws. It adds no
model call. Its content is part of the build fingerprint, so editing it (or
overriding it via `[prompts] build_preamble = "path/to/my_preamble.md"`)
regenerates already-built modules.
