title: Codex Engine
description: "How Jaunt drives the OpenAI Codex CLI (`codex exec`) to generate code."
---

Codex is Jaunt's **sole** generation engine. Every generation command —
`jaunt build`, `jaunt test`, `jaunt skill build`, and `jaunt skill refresh` —
drives the OpenAI **Codex CLI** (`codex exec`) under the hood. Jaunt still owns
discovery, dependency ordering, freshness, validation, repair retries, and
writing the final files into your project tree.

<Callout type="info">
  The older `legacy` (direct provider SDK) and `aider` engines have been
  removed. `agent.engine` must be `"codex"`; any other value is a config error
  (exit code `2`).
</Callout>

## Prerequisites

Codex runs as an external CLI, so install and authenticate it once before
running Jaunt:

```bash
# Install the Codex CLI (see the OpenAI Codex docs for your platform), then:
codex login
```

Jaunt shells out to whatever `codex` is on your `PATH`. Authentication is
handled entirely by Codex (`codex login`, or the `CODEX_API_KEY` env var) —
Jaunt does **not** read `llm.api_key_env` for generation.

## Configure It

`jaunt init` writes a Codex config by default:

```toml
[agent]
engine = "codex"

[codex]
model = "gpt-5.5"
reasoning_effort = "high"   # low | medium | high
sandbox = "workspace-write"
# features = []
# [codex.config]            # raw `-c key=value` passthrough to `codex exec`
```

- `model` — passed to `codex exec --model`.
- `reasoning_effort` — sent as `-c model_reasoning_effort=...`.
- `sandbox` — the Codex sandbox policy (`codex exec --sandbox`). Generation
  uses your configured value; Jaunt pins internal read-only steps to
  `read-only`.
- `features` / `[codex.config]` — advanced raw passthrough to the Codex CLI.

## How A Generation Call Works

For each module Jaunt needs to generate:

1. Jaunt renders the prompt (Jaunt preamble + system template + your spec's
   source segment + optional per-spec extras and the repo map) and seeds
   builtin/project skills into the workspace at `.agents/skills/` for Codex to
   discover natively.
2. It spawns one `codex exec --json` subprocess and passes the prompt on stdin.
3. Codex edits inside the configured sandbox; Jaunt parses the JSONL event
   stream for the final result.
4. Jaunt validates the generated module (AST + type checks) and, on failure,
   runs bounded repair retries before writing the file into `__generated__/`.

Each call is its own subprocess, so generation is naturally task-local and
parallel-safe — there are no shared credentials and no process-wide locks.

## Freshness

Jaunt fingerprints the Codex runtime as part of artifact freshness. Changing
any of these marks generated files stale, so they regenerate without `--force`:

- `agent.engine`
- `codex.model`
- `codex.sandbox`
- `codex.reasoning_effort`
- the rendered prompt template content

## Test Generation Style

Generated pytest modules must satisfy the literal test specs you wrote. By
default Jaunt tests through the **public module contract**, not generated or
wrapper internals, unless a spec explicitly opts out. Spell out every required
behavior in the spec rather than relying on the engine to infer a broad,
unstated test matrix.
