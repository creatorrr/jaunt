---
title: Adding Jaunt To Your Project
description: "Practical setup for a real repo (config, where specs live, and how to ship generated code)."

This guide is for using Jaunt as a tool inside an existing Python repo, not for hacking on the Jaunt repo itself.

## 1) Install Jaunt

First, install the OpenAI **Codex CLI** (Jaunt's generation engine) and
authenticate it:

```bash
codex login
```

Then install Jaunt however you vend it (local path, internal index, or git).
You’ll also want pytest for `jaunt test`.

Examples:

```bash
# Package index:
uv add jaunt

# Local checkout (editable):
uv add --editable /path/to/jaunt
```

There are no provider SDK extras to install — Codex handles its own
authentication (`codex login`, or `CODEX_API_KEY`).

## 2) Add `jaunt.toml`

At your project root:

```toml
version = 1

[paths]
source_roots = ["src"]
test_roots = ["tests"]
generated_dir = "__generated__"
```

<Callout type="info">
  **Choosing `source_roots`.** Point a root at the *parent* of your package
  (e.g. `src`, so specs are `my_app.specs`), not at the package directory
  itself. A root that is itself a package (contains `__init__.py`) yields
  bare module names like `timing` that can shadow stdlib/PyPI names — Jaunt
  warns when it detects this. Unknown keys or sections in `jaunt.toml` are
  now rejected with an error (exit 2), so run `jaunt instructions` to print
  the full annotated schema if you are unsure.
</Callout>

## 3) Pick A Good First Spec

Best first targets:

- validators
- parsers
- formatters
- glue code (boring, correctness-heavy utilities)

Avoid (for now):

- high-performance hot paths
- complex stateful systems with tight invariants

## 4) Write Spec Stubs In Your Normal Source Tree

Put `@jaunt.magic` specs in real modules under your package, next to the code that will call them. Treat them as your public API.

## 5) Run `jaunt build`

```bash
jaunt build
```

## 6) Import And Use The Spec Module

You import the spec module, not the generated module. The decorator forwards calls into `__generated__/`.

## Coverage Gates

Spec stub bodies are unreachable by design — the runtime forwards every call
into `__generated__/`, so the `raise RuntimeError("spec stub …")` line in a
`@jaunt.magic` stub never executes. Before wiring `--cov-fail-under` on a
converted module, exclude the stub-raise line from coverage:

```toml
[tool.coverage.report]
exclude_lines = ["raise RuntimeError\\(\"spec stub", "pragma: no cover"]
```

## 7) Generated Code In Git: Commit Or Ignore?

Two reasonable workflows:

1. Commit generated code.
2. Ignore generated code.

Commit generated code:

- Deterministic installs. CI doesn’t need LLM access.
- You’ll see generated diffs in PRs.

Ignore generated code:

- Cleaner repo. Output stays disposable.
- CI must run generation (so you need API keys in automation).

Pick one and be consistent.

## Migrating To 1.3.0

If you adopted an earlier Jaunt and use **top-level** spec modules, run
`jaunt clean && jaunt build` once — top-level modules now generate to
`__generated__/<module>.py` instead of `<module>/__generated__/__init__.py`.
Unknown `jaunt.toml` sections/keys now error (exit 2). `jaunt check` now gates
`@jaunt.magic` freshness in addition to contract batteries, so it is the single
CI command for both modes.
