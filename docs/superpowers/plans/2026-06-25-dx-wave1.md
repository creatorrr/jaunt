# Jaunt DX Wave 1 + skills — Implementation Plan

Driven by the Ultimate-Tic-Tac-Toe dogfood (`examples/07_ultimate_ttt/CASE-STUDY.md`) and a
grounded Codex review (thread `019efd68`, read-only over the repo). Executed via an
Opus-subagent workflow driving `codex exec`. Branch `feat/codex-engine`.

## How this is executed
Each task is one Opus supervisor that DIRECTS `codex exec` to write the code, REVIEWS the
diff (revert scope creep / weakened constraints), VERIFIES, and COMMITS. Tasks run strictly
**sequentially** (they share `cli.py`, `codex_backend.py`, `builder.py`).

## Global constraints
- Dev-time codex call: `codex exec --dangerously-bypass-approvals-and-sandbox -c model_reasoning_effort="high" - < PROMPTFILE` (prompt on stdin; prompt file OUTSIDE the repo).
- Green gate per task (from repo root): `uv run pytest -q && uv run ruff check . && uv run ty check`. Commit ONLY when all green.
- Do NOT stage `examples/06_whole_class/tests/` (pre-existing untracked) or temp prompt files. Do NOT push.
- If a task can't reach green, STOP and report — don't commit red.

## Task 1 — Truthful Codex failure reporting + cached tokens (P0)
**Files:** `src/jaunt/generate/codex_backend.py`, `tests/test_codex_backend.py`
**Why:** today a failed `codex exec` is swallowed — the backend reads back the unmodified
seed, so a real infra/model failure masquerades as a "method left as a stub" validation error.
**Do:**
- In `run_codex_exec`/`generate_module`, detect a **real exec failure** with this layered rule
  (Codex review): the `--json` JSONL stream contains a `turn.failed` event **or** a top-level
  `error` event; **or** subprocess `returncode != 0`; **or** the stream never emitted
  `turn.completed` (protocol failure). On any of these, raise a clear error carrying Codex's
  stderr / the error item's message, so the builder reports the REAL failure.
- **Do NOT** treat "target file unchanged from the seed" as an exec failure — a completed turn
  may legitimately write identical or low-quality content; that stays a validation concern.
- Parse `cached_input_tokens` from `turn.completed` into `TokenUsage.cached_prompt_tokens`
  (currently dropped).
**Tests:** non-zero exit; `turn.failed`; top-level `error`; missing `turn.completed`; cached-token parsing.
**Commit:** `fix(codex): surface real codex exec failures + parse cached tokens`

## Task 2 — Retry once on model-config errors (depends on Task 1)
**Files:** `src/jaunt/generate/codex_backend.py`, `tests/test_codex_backend.py`
**Do:** using Task 1's failure parser, when the failure is a model-config error (e.g.
verbosity unsupported for the model), retry the call ONCE without the offending
`[codex.config]` key (or a safe value). Must not become a silent retry loop.
**Tests:** a simulated config error triggers exactly one retry; a non-config failure does not.
**Commit:** `feat(codex): retry once on model-config errors (e.g. verbosity)`

## Task 3 — Dual-form decorators (`@magic`/`@test`)

**Tests:** bare `@magic`, bare `@test`, called forms still work, contract bare+called.
**Commit:** `fix(runtime): @jaunt.magic/@test accept bare and called forms`

## Task 7 — Stale-cascade fixes (two real bugs)
**Files:** `src/jaunt/cli.py` (cmd_build target handling), `src/jaunt/builder.py` (run_build
expansion), `src/jaunt/generate/fingerprint.py`, tests
**Do (per Codex review):**
- **Target closure:** when `--target` is set, constrain `run_build`'s staleness expansion to
  the requested target's transitive closure — don't pull in dependents OUTSIDE it just because
  an allowed upstream dep shows API drift.
- **Fingerprint:** under the Codex engine, the generation fingerprint must NOT depend on the

  them currently stales artifacts spuriously. Drop them from the codex fingerprint (or replace
  with the actual codex prompt inputs).
**Tests:** targeted build does not rebuild modules outside the target closure; editing an
unused prompt template does not stale codex-engine artifacts.
**Commit:** `fix(builder,cli): targeted build stays in closure; codex fingerprint drops unused templates`

## Task 8 — Auto-skills slimming (kill switch + caps + selective injection)

`src/jaunt/skill_manager.py`, `src/jaunt/config.py`, `src/jaunt/cli.py`, tests
**Do (per Codex review — auto-skills inflate the ~120K-token prompt):**

- Hard-cap injected skill text per dist (well below the current 50K README / 100K source

**Tests:** `auto = false` disables auto-skill injection; per-dist text is capped; only relevant
skills are injected into a given module's prompt.
**Commit:** `feat(skills): auto-skills kill switch + caps + selective injection (slim prompts)`
